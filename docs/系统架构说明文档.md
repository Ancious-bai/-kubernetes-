# 基于 Kubernetes 的 YOLO 矸石识别训练平台 — 系统架构说明文档

## 一、系统概述

本系统是一个基于 Kubernetes 集群的 YOLO 矸石识别模型分布式训练平台，旨在实现从数据上传、预处理、模型训练、模型测试到模型推理的全流程自动化管理。系统采用前后端分离架构，后端基于 Spring Boot 构建，前端基于 Vue 3 构建，通过 Kubernetes API 实现训练任务的调度与资源管理，通过 NFS 共享存储实现多节点间的数据一致性。

### 1.1 核心能力

- **数据管理**：支持ZIP压缩包上传、自动解压、YOLO格式预处理
- **训练调度**：FIFO队列调度、动态节点选择、资源预估与不足警告
- **资源管理**：动态并发计算、节点cordon/uncordon控制、实时资源监控
- **模型管理**：模型库、模型推理、推理结果可视化
- **用户管理**：三级角色（ROOT/ADMIN/USER）、admin隔离、角色不可变
- **实时监控**：WebSocket实时日志、训练进度追踪、节点训练历史

### 1.2 技术栈

| 层级 | 技术选型 | 版本 |
|------|---------|------|
| 前端 | Vue 3 + Element Plus + Vite | Vue 3.4 / Element Plus 2.x |
| 后端 | Spring Boot + Spring Security + JPA | 3.x / JDK 21 |
| 数据库 | MySQL | 8.0 |
| 容器编排 | Kubernetes | 1.28+ |
| 容器运行时 | Docker / containerd | 24.x |
| 共享存储 | NFS + Kubernetes PVC | - |
| GPU支持 | NVIDIA Device Plugin | - |
| 认证 | JWT (HS256) | - |

## 三、核心模块架构

### 3.1 训练调度器架构

训练调度器是系统的核心模块，采用**生产者-消费者模式**，由`TrainingSchedulerService`实现：

```
用户请求 → addTask() → PriorityBlockingQueue(FIFO) → processQueue() → executeTask()
                │                                              │
                │ 创建TrainingRecord(QUEUED)                    │ 创建K8s Job
                │ 记录NodeLog(QUEUED)                           │ 轮询Pod状态
                │                                               │ 更新进度
                ▼                                               ▼
           数据库持久化                                    K8s集群执行
```

**关键设计决策**：

1. **延迟节点选择**：任务入队时不选择节点，在`processQueue()`实际调度时才根据最新负载动态选择，避免负载变化导致不均衡
2. **FIFO调度**：按提交时间排序，无优先级，防止高优先级任务持续抢占资源
3. **综合评分选节点**：基于剩余CPU、内存、GPU、并发槽位的加权评分，选择资源最充裕的节点
4. **动态资源分配**：根据数据集大小自动计算训练Job的CPU/内存请求和限制

### 3.2 节点管理架构

```
K8s API Server
      │
      │ listNodes() / listPods()
      ▼
K8sClientService.getNodeInfoList()    K8sClientService.getNodeAllocatedResources()
      │                                        │
      │ 节点静态信息                              │ 节点动态资源
      │ (名称/IP/CPU/内存/GPU/角色)               │ (已请求CPU/内存/GPU/运行Pod数)
      ▼                                        ▼
NodeManagementService.syncNodesFromCluster()  ←── 每30秒自动执行
      │
      ├── updateNodeFromCluster()     更新数据库
      ├── calculateDynamicConcurrent() 计算并发数
      └── 删除已移除的节点               清理数据库
```

**动态并发算法**：
- 可用CPU = allocatable - 系统预留(Master:2核, Worker:1核)
- 可用内存 = allocatable - 系统预留(Master:2GB, Worker:1GB)
- 并发 = min(可用CPU/1, 可用内存/1GB, GPU数量, Master最多1)
- 最低保证1个并发

### 3.3 用户权限架构

```
ROOT (最高权限)
 ├── 查看所有数据（数据集/训练记录/用户/日志/节点）
 ├── 创建ADMIN和USER
 ├── 删除任何非ROOT用户
 ├── 清空全部数据
 └── 删除节点记录

ADMIN (管理员，有隔离)
 ├── 只看自己+自己创建的user的数据
 ├── 只能创建USER
 ├── 只能删除自己创建的USER
 ├── 查看节点管理（IP/调度/并发）
 └── 操作节点（cordon/uncordon）

USER (普通用户)
 ├── 只看自己的数据集和训练记录
 ├── 上传数据集、提交训练
 ├── 查看节点基本信息（隐藏IP/调度/并发/操作）
 ├── 查看模型库（自己的）
 └── 修改自己的密码
```

**关键设计**：
- 角色一旦创建不可更改，防止权限提升
- `createdBy`字段建立用户归属关系，admin之间完全隔离
- 数据集、训练记录、模型库、操作日志均按`createdBy`过滤

## 四、数据流架构

### 4.1 训练全流程数据流

```
1. 上传数据集
   浏览器 → POST /api/datasets/upload → YoloController
   → 保存ZIP到NFS → 解压 → datasets表记录

2. 数据预处理
   浏览器 → POST /api/preprocess → YoloController
   → K8sClientService.buildPreprocessJob() → K8s创建Job
   → Job在节点上执行python预处理脚本 → 生成data.yaml和train/val分割

3. 提交训练
   浏览器 → POST /api/scheduler/estimate → 资源预估确认
   → POST /api/scheduler/add → TrainingSchedulerService.addTask()
   → training_records表(QUEUED) → node_logs表(QUEUED) → FIFO队列

4. 调度执行
   processQueue() → selectNodeForTask() → 动态选择节点
   → YoloService.startTrainingOnNode() → calculateDynamicResources()
   → K8sClientService.buildTrainingJob(nodeName, cpuRequest, memRequest, ...)
   → K8s创建Job → Pod在目标节点启动 → 执行YOLO训练

5. 实时监控
   K8s Pod日志 → LogWebSocketHandler → WebSocket → 浏览器实时显示
   K8s Pod状态 → YoloService轮询 → 更新training_records进度

6. 训练完成
   training_records(COMPLETED) → node_logs(RUNNING→COMPLETED)
   → 日志持久化到NFS → runs目录生成results.csv和weights

7. 查看结果
   浏览器 → GET /api/nodes/runs/{recordName}/train → 读取results.csv和文件列表
   → GET /api/nodes/runs/{recordName}/train/file?path=xxx → 读取具体文件内容

8. 保存模型
   浏览器 → POST /api/models → 复制best.pt到saved_models → model_library表记录

9. 模型推理
   浏览器 → POST /api/models/{id}/predict → K8s创建推理Job
   → 生成推理结果图片 → GET /api/models/{id}/predict-image → 浏览器展示
```

### 4.2 删除级联数据流

```
删除数据集(dataName):
  datasets表 → training_records表(按dataName) → node_logs表(按recordName)
  → model_library表(按dataName) + .pt文件 → K8s Job/Pod → NFS目录

删除训练记录(recordName):
  training_records表 → node_logs表(按recordName) → model_library表(按recordName) + .pt文件
  → K8s Job/Pod → NFS runs/logs目录

清空全部:
  datasets + training_records + node_logs + model_library + operation_logs
  + K8s所有训练Job/Pod + NFS所有数据目录
```

## 六、安全架构

### 6.1 认证流程

```
1. 用户登录 → POST /api/auth/login → 验证密码 → 生成JWT令牌
2. 后续请求 → Authorization: Bearer <token> → JwtAuthFilter拦截验证
3. 特殊场景 → URL参数 ?token=<token> → JwtAuthFilter也支持（用于img标签）
4. WebSocket → 连接时通过URL参数传递token → WebSocketConfig验证
```

### 6.2 权限控制

- **接口级**：Controller方法中通过`httpRequest.getAttribute("role")`获取角色，手动判断权限
- **数据级**：查询时根据`createdBy`字段过滤，确保admin隔离和user隔离
- **操作级**：删除操作检查是否为创建者或ROOT

### 6.3 K8s RBAC

后端使用`yolo-scheduler` ServiceAccount，拥有以下集群级权限：
- `batch/jobs`: 完整CRUD + delete
- `pods, pods/log`: get, list, watch, delete
- `nodes`: get, list, watch, **patch**（cordon/uncordon需要）
- `persistentvolumeclaims`: get, list, watch, create, delete
- `configmaps, secrets`: get, list

## 七、容错与高可用

### 7.1 当前容错机制

| 故障场景 | 当前行为 | 影响 |
|---------|---------|------|
| Worker节点宕机 | 该节点Pod标记为Unknown，K8s不会自动迁移 | 该节点训练任务中断，需手动重新提交 |
| Master节点宕机 | 所有控制平面组件停止 | 整个集群不可用，训练任务继续运行但无法管理 |
| 后端Pod崩溃 | K8s Deployment自动重启 | 服务短暂中断（约30秒），无状态设计确保数据不丢失 |
| MySQL Pod崩溃 | K8s StatefulSet自动重启 | 数据持久化在PVC，重启后数据恢复 |
| NFS不可用 | 所有依赖NFS的Pod I/O挂起 | 训练任务无法读写数据 |

### 7.2 高可用建议

1. **后端多副本**：设置`replicas: 2`，配合Pod反亲和性分散到不同节点
2. **MySQL主从**：使用MySQL主从复制，避免单点故障
3. **NFS高可用**：使用分布式存储（如Ceph）替代单节点NFS
4. **Master高可用**：部署3个Master节点组成HA集群
