# 三节点集群配置信息说明文档

## 一、集群拓扑概览

本系统采用一主两从（1 Master + 2 Worker）的 Kubernetes 集群架构，三个节点均为 Ubuntu 虚拟机，通过 kubelet 注册到同一集群。所有节点通过内网互通，NFS 共享存储确保训练数据在节点间一致可见。

| 节点名称 | 角色 | 职责 | 运行服务 |
|---------|------|------|---------|
| master | 控制平面 + 工作节点 | K8s管理 + 平台服务 + 少量训练 | kube-apiserver, kube-controller-manager, kube-scheduler, etcd, MySQL, 后端, 前端, NFS Provisioner |
| worker1 | 工作节点 | GPU训练 | kubelet, NFS客户端, 训练Job |
| worker2 | 工作节点 | GPU训练 | kubelet, NFS客户端, 训练Job |

## 二、Master 节点详细配置

### 2.1 硬件资源配置

| 资源类型 | 总量(Capacity) | 可分配(Allocatable) | 系统预留 |
|---------|---------------|-------------------|---------|
| CPU | 2核（可调） | 约1.8核 | K8s预留约100m |
| 内存 | 4GB（可调） | 约3.5Gi | K8s预留约500Mi |
| GPU | 0 | 0 | 无GPU设备 |

### 2.2 运行的K8s系统组件

| 组件 | 资源请求 | 资源限制 | 说明 |
|------|---------|---------|------|
| kube-apiserver | 250m CPU / 512Mi | 无限制 | K8s API入口，所有kubectl和客户端调用都经过此组件 |
| kube-controller-manager | 200m CPU / 256Mi | 无限制 | 管理Pod副本数、节点状态等控制器 |
| kube-scheduler | 100m CPU / 128Mi | 无限制 | K8s默认调度器（本系统训练任务通过nodeName绕过） |
| etcd | 100m CPU / 256Mi | 无限制 | 集群状态存储，所有K8s资源定义的持久化数据库 |
| kube-proxy | 100m CPU / 64Mi | 无限制 | Service网络代理，负责Service到Pod的流量转发 |
| CoreDNS | 100m CPU / 70Mi | 2 CPU / 170Mi | 集群内DNS解析 |

### 2.3 运行的平台服务

| 服务 | 部署方式 | 资源请求 | 资源限制 | 持久化 |
|------|---------|---------|---------|--------|
| yolo-mysql | StatefulSet | 250m CPU / 512Mi | 1 CPU / 1Gi | 10Gi PVC (RWO) |
| yolo-backend | Deployment | 500m CPU / 512Mi | 2 CPU / 1Gi | 无状态，数据在MySQL和NFS |
| yolo-frontend | Deployment | 100m CPU / 128Mi | 500m CPU / 256Mi | 无状态，静态文件 |
| nfs-provisioner | Deployment | 未设置 | 未设置 | 动态创建NFS PV |

### 2.4 Master节点资源分配分析

Master节点总可分配约1.8核CPU和3.5Gi内存。系统组件和平台服务已占用约1.4核CPU和1.6Gi内存，剩余约0.4核CPU和1.9Gi内存。由于Master节点承载了控制平面和平台服务，动态并发计算算法将Master的最大并发任务数限制为1，且仅在Worker节点资源不足时才调度训练任务到Master。

### 2.5 Master节点特殊限制

- **并发上限**：动态计算为1（`calculateDynamicConcurrent`中`isMaster`分支限制）
- **系统预留**：2核CPU + 2GB内存（比Worker多预留1核1GB，保护控制平面稳定性）
- **调度优先级**：最低，仅在Worker节点无可用资源时作为兜底

## 三、Worker1 节点详细配置

### 3.1 硬件资源配置

| 资源类型 | 总量(Capacity) | 可分配(Allocatable) | 系统预留 |
|---------|---------------|-------------------|---------|
| CPU | 4核（可调） | 约3.9核 | K8s预留约100m |
| 内存 | 8GB（可调） | 约7.5Gi | K8s预留约500Mi |
| GPU | 1（nvidia.com/gpu） | 1 | 无预留 |

### 3.2 运行的系统组件

| 组件 | 资源请求 | 说明 |
|------|---------|------|
| kubelet | 约100m CPU / 200Mi | 节点代理，管理Pod生命周期 |
| kube-proxy | 100m CPU / 64Mi | Service网络代理 |
| nvidia-device-plugin | 约50m CPU / 30Mi | GPU设备插件，向K8s注册GPU资源 |

### 3.3 动态并发计算过程

以Worker1为例，`calculateDynamicConcurrent`的计算过程：

1. **CPU核数**：4核，系统预留1核 → 可用3核
2. **内存**：7680Mi（约7.5Gi），系统预留1GB → 可用约6.5Gi（6656Mi）
3. **CPU并发**：3核 / 1核每任务 = 3
4. **内存并发**：6656Mi / 1024Mi每任务 = 6
5. **取较小值**：min(3, 6) = 3
6. **GPU限制**：1个GPU → min(3, 1) = 1
7. **最终并发**：max(1, 1) = **1**

**关键限制**：虽然CPU和内存可以支持3个并发任务，但GPU数量为1，YOLO训练必须使用GPU，因此实际并发上限为1。如果未来需要多并发，需要增加GPU数量。

### 3.4 Worker1资源分配分析

Worker1可分配约3.9核CPU和7.5Gi内存。系统组件占用约250m CPU和300Mi内存，剩余约3.65核CPU和7.2Gi内存。一个训练Job请求500m CPU和1Gi内存，理论上CPU可承载7个任务、内存可承载7个任务，但受GPU限制只能1个并发。

## 四、Worker2 节点详细配置

### 4.1 硬件资源配置

| 资源类型 | 总量(Capacity) | 可分配(Allocatable) | 系统预留 |
|---------|---------------|-------------------|---------|
| CPU | 3核（可调） | 约2.9核 | K8s预留约100m |
| 内存 | 4GB（可调） | 约3.5Gi | K8s预留约500Mi |
| GPU | 1（nvidia.com/gpu） | 1 | 无预留 |

### 4.2 动态并发计算过程

1. **CPU核数**：3核，系统预留1核 → 可用2核
2. **内存**：3584Mi（约3.5Gi），系统预留1GB → 可用约2.5Gi（2560Mi）
3. **CPU并发**：2核 / 1核每任务 = 2
4. **内存并发**：2560Mi / 1024Mi每任务 = 2
5. **取较小值**：min(2, 2) = 2
6. **GPU限制**：1个GPU → min(2, 1) = 1
7. **最终并发**：max(1, 1) = **1**

### 4.3 Worker2与Worker1对比

| 维度 | Worker1 | Worker2 |
|------|---------|---------|
| CPU | 4核 | 3核 |
| 内存 | 8GB | 4GB |
| GPU | 1 | 1 |
| 动态并发 | 1 | 1 |
| 训练性能 | 更快（CPU/内存充裕） | 较慢（资源紧张） |

Worker2的CPU和内存较少，但GPU数量与Worker1相同。在GPU训练场景下，两个Worker的并发上限相同（均为1），但Worker1在数据预处理和模型推理等CPU密集型任务上表现更好。

## 五、NFS共享存储配置

### 5.1 NFS服务器

NFS服务运行在Master节点上，通过nfs-subdir-external-provisioner提供动态PV供给。

| 配置项 | 值 |
|--------|-----|
| NFS服务器地址 | master节点IP（如192.168.x.x） |
| NFS共享路径 | /srv/nfs/k8s |
| StorageClass名称 | nfs-storage |
| 回收策略 | Retain |
| 访问模式 | ReadWriteMany (RWX) |

### 5.2 PVC配置

| PVC名称 | 容量 | 访问模式 | 用途 |
|---------|------|---------|------|
| yolo-workspace | 50Gi | ReadWriteMany | 训练数据、模型、日志共享目录 |
| mysql-data | 10Gi | ReadWriteOnce | MySQL数据持久化 |

## 六、网络配置

### 6.1 Service网络

| Service名称 | 类型 | 端口映射 | 说明 |
|-------------|------|---------|------|
| yolo-frontend | NodePort | 30080:80 | 前端访问入口 |
| yolo-backend | ClusterIP | 8080 | 后端API（集群内访问） |
| yolo-mysql | ClusterIP | 3306 | MySQL（集群内访问） |

### 6.2 外部访问

- **前端**：通过 `http://<任意节点IP>:30080` 访问
- **后端API**：前端通过Nginx反向代理访问后端，路径`/api/*`转发到后端Service
- **WebSocket**：路径`/ws/*`通过Nginx的`proxy_set_header Upgrade`配置支持

## 七、资源动态调整说明

### 7.1 调整虚拟机配置后的生效流程

1. 关闭虚拟机，在虚拟化软件中调整CPU/内存
2. 启动虚拟机，kubelet自动检测新硬件资源并上报给K8s API Server
3. 后端启动5秒后自动执行`syncNodesFromCluster()`，从K8s获取最新资源数据
4. 每30秒定时同步一次，确保数据实时性
5. `calculateDynamicConcurrent()`根据新资源重新计算并发上限

### 7.2 节点移除后的处理

当节点从K8s集群中移除（如`kubectl delete node worker2`）：
1. 下次同步时（30秒内），`syncNodesFromCluster()`检测到该节点不在集群中
2. 自动从数据库中删除该节点记录
3. 前端不再显示该节点
4. 该节点上正在运行的训练任务不受影响（已提交的Job继续运行）

### 7.3 GPU资源扩展

如果需要增加并发训练数，需要：
1. 在Worker节点上安装更多GPU
2. 安装nvidia-device-plugin并重启kubelet
3. K8s自动识别新GPU，`gpuAllocatable`字段更新
4. 动态并发算法自动根据GPU数量调整并发上限
